## Evolution loop phase sequence

The `EvolutionLoop` (`runtime/evolution/evolution_loop.py`) is the core autonomous execution engine.
As of v4.0.0 it runs a **15-phase governance-gated sequence** per epoch:

```
Phase 0a  Governance gate check       — GovernanceGate pre-flight
Phase 0b  E/E mode selection          — ExploreExploitController.select_mode()
Phase 1   Mutation proposal           — AIMutationProposer (Claude API)
Phase 2   Fitness scoring             — FitnessOrchestrator.score()
Phase 3   Governance approval         — GovernanceGate approval gate
Phase 4   Weight adaptation           — WeightAdaptor update
Phase 4b  Promotion regression gate   — PolicyPromotionController → rollback if degraded  [PR-10-02]
Phase 5   Landscape recording         — FitnessLandscape win/loss
Phase 5b  E/E commit                  — ExploreExploitController.commit_epoch()
Phase 5c  Craft pattern extraction    — CraftPatternExtractor → SoulboundLedger           [PR-9-02]
Phase 5d  Reward signal ingestion     — RewardSignalBridge → reward_learning pipeline     [PR-10-01]
Phase 6   Checkpoint                  — state persistence and journal flush
```

All phases are exception-isolated. Phase 4b triggers automatic rollback to `last_promoted_weights`
on detected regression. Phase 5d bridges `MutationScore` results into the reward learning pipeline
and persists a `fitness_signal` entry to the tamper-evident SoulboundLedger.

## Governance & Determinism Guarantees (Current State)

The following invariants are enforced at all times and are verified by the CI governance gate:

**Constitution v0.3.0 BLOCKING rules:**
- `governance_health_floor` — epochs below health floor are rejected; no mutation applied
- `soulbound_privacy_invariant` — private key material cannot appear in ledger payloads (added v3.4.0)
- `bandit_arm_integrity_invariant` — arm stats non-negative; consecutive agent lock-in capped (added v3.6.0)
- `market_signal_integrity_invariant` — blocks weight promotion when consecutive synthetic market epochs exceed cap (added v3.8.0)

**Determinism invariants:**
- Task normalization is deterministic (first-seen order, deduplicated labels)
- Replay produces byte-identical mutation proposals for the same epoch seed
- `ADAAD_DETERMINISTIC_LOCK=1` enforces strict determinism mode at runtime
- Empty/whitespace-only task batches resolve to `safe_boot=True`

**Environment controls (operator-configurable):**

| Variable | Default | Effect |
|---|---|---|
| `ADAAD_DETERMINISTIC_LOCK` | `''` | `1\|true\|yes\|on` enables strict determinism mode |
| `ADAAD_DISPATCH_LATENCY_BUDGET_MS` | `6.0.0` | Maximum dispatch latency budget in milliseconds |
| `ADAAD_SOULBOUND_KEY` | required | HMAC-ENV key ≥32 bytes; absence is fail-closed |
| `ADAAD_ROADMAP_AMENDMENT_TRIGGER_INTERVAL` | — | Epoch interval for roadmap amendment checks |

> Full variable reference: [docs/ENVIRONMENT_VARIABLES.md](ENVIRONMENT_VARIABLES.md)

**Fail-closed surfaces:**
- Missing `ADAAD_SOULBOUND_KEY` → `SoulboundKeyError` raised; `soulbound_key_absent.v1` journaled; system halts
- GovernanceGate rejection → epoch aborted; no mutation applied; event journaled
- PolicyPromotionController regression → rollback to `last_promoted_weights`; epoch continues with prior profile

## Orchestration hardening invariants

`app/orchestration/mutation_orchestration_service.py` enforces:

- Task normalization is deterministic (first-seen order, deduplicated labels).
- Empty/whitespace-only task batches resolve to `safe_boot=True`.
- Malformed/non-string task records are ignored without raising side effects.
- Transition failures preserve fail-closed behavior (`run_cycle=False`).
- Blocked transition payloads are isolated per envelope to prevent cross-call mutation bleed.
