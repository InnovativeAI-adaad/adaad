## Summary

v9.12.1 closes two post-Phase-77 outstanding items surfaced by PR #508: the
`lineage_v2.py` warm-cache O(n→n²) regression, and the autonomous doc-sync
infrastructure that accumulated documentation drift (placeholder infobox,
missing CHANGELOG entries for Phases 77+ and the optimize sweep) during the
Phase 76→77 transition. This release also formally registers Phase 78 on the
ROADMAP.

## What Shipped

### Runtime Fix — Warm-Cache O(n→n²) Regression (`lineage_v2.py`)

- **Bug:** `append_event()` nullified `_verified_tail_hash` after every write,
  forcing a full O(n) re-scan on every subsequent `_last_hash()` call.
  Repeated appends over ledger lifetime → O(n²) total scan cost.
- **Fix:** Advance `_verified_tail_hash` to `entry["hash"]` post-append.
  Consistent with C-04 warm-cache contract.
- **Evidence ref:** `optimize-warm-cache-lineage-v2`

### Runtime Fix — Governance Event Wiring (`app/github_app.py`, `app/main.py`)

- `app/github_app._emit_governance_event` wired correctly to
  `ExternalEventBridge` keyword API (Phase 77 FINDING-AUDIT-C03 closure).
- `app/main.py` import contract aligned to `runtime.api` facade (5 modules).
- **Evidence refs:** `optimize-github-app-wiring`, `optimize-import-contracts`

### Test Suite — Constitution Version Constant Sync

- `tests/determinism/test_replay_attestation_determinism.py` — monkeypatch

## Phase 78 Deferred Items

Journal-level `_VERIFIED_TAIL_CACHE` (`security/ledger/journal.py`) was
benchmarked at **11.6× speedup** (1,700 ms → 146 ms per `evaluate_mutation`)
during this sweep but deferred: a pre-existing shared-journal test-isolation
issue was exposed under load. Phase 78 M78-01 resolves isolation first, then
enables the cache. No user-facing performance regression in this release.
