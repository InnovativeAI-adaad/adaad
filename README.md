CI is a quality gate on output. ADAAD is a governance framework on the mutation process. Every change has a ledger entry with a hash-chain link to the prior state, a fitness score, a gate decision, and a deterministic replay proof. You can delete your CI history. You cannot alter ADAAD's ledger.
</details>

<details>
<summary><b>What is the Seed Lifecycle Pipeline?</b></summary>
<br/>

ADAAD generates `CapabilitySeed` objects — scored proposals for its own capability improvements. Seeds with `expansion_score ≥ 0.85` enter the `SeedPromotionQueue`. A human governor reviews and approves them. Approved seeds become `ProposalRequest` objects, get injected into a Constitutional Evolution Loop epoch, and have their outcomes recorded in the lineage ledger.

`SEED-LIFECYCLE-COMPLETE-0` — the full provenance chain from seed inception to CEL outcome is hash-linked — was demonstrated live on March 20, 2026. ADAAD can now propose and evaluate its own capability evolutions under constitutional governance.
</details>

<details>
<summary><b>What is the Multi-Generation Lineage Graph?</b></summary>
<br/>

Phase 79 added `MultiGenLineageGraph` — a directed acyclic graph that tracks parent→child relationships between governed epochs. Given a `LineageLedgerV2`, it reconstructs the full evolution ancestry: which epochs seeded which, how many generations deep the evolution tree is, and which nodes are seed-driven vs autonomously generated.

`MULTIGEN-REPLAY-0`: the graph is fully reconstructable from the ledger alone. `MULTIGEN-DETERM-0`: identical ledger contents produce identical `graph_digest`. This is the foundation for Phase 80 compound evolution — epochs that explicitly build on the mutations of prior epochs.
</details>

<details>
<summary><b>What governance guarantees does Ed25519 signing add?</b></summary>
<br/>

The current ledger uses SHA-256 hash chaining: each entry's hash includes the previous entry's hash. This makes substitution detectable (breaking the chain) but does not prevent an attacker with write access from appending new entries.

Phase 78 implemented `Ed25519FileSigner` and `HMACEnvSigner` — production-ready `EventSigner` implementations backed by env-configurable keys. Phase 80 wires these to `LineageLedgerV2.append_event()` so every entry carries a `SignatureBundle`. After Phase 80, appending entries requires access to the signing key. An attacker without the key cannot produce valid signatures. `verify_integrity()` detects both hash-chain breaks and invalid signatures.
</details>

Because constitutional governance should not require enterprise infrastructure. The entire runtime — CEL, ledger, sandbox, Aponi dashboard — is pure Python with no binary dependencies unavailable on Android.

Running on constrained hardware is also a correctness argument: if your "governed" system only operates safely with AWS KMS and Kubernetes, the safety properties are not yours — they are on loan from your cloud provider. ADAAD's safety properties are computational: they derive from SHA-256 hash chains and the Python runtime. Available on any hardware.
</details>
